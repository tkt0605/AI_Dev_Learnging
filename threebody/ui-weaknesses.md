# ThreeBody UI — 弱みリスト

> 分析軸: 視覚・体験・パラダイム  
> 分析時点のブランチ: `feature/edit_triangle_UI`

---

## 視覚（Visual）

### W-V1: ゼロ状態に視覚的インパクトがない
初期表示は白い点が一つあるだけの暗い画面。三角形の美しさはノードを3つ置いて初めて現れるが、ユーザーはその状態を見るまで魅力に気づけない。

```
初期画面:
  [白い点]
  ← opacity: 0.18 のヒントテキストのみ
```

**影響:** 初見ユーザーが何をすべきか分からず離脱するリスクが高い。

---

### W-V2: サイドバーのパレットが地味
カラードットとラベルの一覧に留まっており、「ここからドラッグして三角形を作る」というアクションを視覚的に誘導できていない。

---

### W-V3: ノードが円のみで個性が弱い
各ノード（Chat・MCP・Context）は色違いの円。アイコンはあるが形状の差異がないため、三者の「役割の違い」が視覚的に伝わりにくい。

---

## 体験（UX）

### W-U1: 三角形の配置がAIの動作に影響しない 【最重要】
ノードを置いても置かなくても、どの位置に置いても Chat の挙動は変わらない。「三角形を組む」という操作がシステムの動作に繋がっていないため、体験の核心が空洞になっている。

```ts
// useChat.ts — 現状はモック
function simulateStream(id: string, text: string) {
  // 実際のAI呼び出しなし。エコーバックするだけ
  block.content += text[i]
}
```

**影響:** ユーザーが「三角形を作ること」の意味を理解できない。

---

### W-U2: Chat が完全にモック
`sendMessage` は入力テキストをオウム返しするだけで、実際のAI応答は返ってこない。ユーザーが試用すると即座に「動いていない」と気づく。

---

### W-U3: Build ボタンが何もしない
1.2秒のスピナーアニメーションの後「完了」と表示されるが、実際には何もビルドされない。

```ts
// AppAside.vue
setTimeout(() => {
  building.value = false
  built.value = true  // ← 見た目だけ「完了」
}, 1200)
```

**影響:** ユーザーの期待を裏切り、UIへの信頼を損なう。

---

### W-U4: Context ダイアログがキャンセルできない
`SettingsDialog` は draft パターン（編集中はローカルコピー、保存ボタンで確定）を採用しているが、`ContextDialog` は `settings.systemPrompt` に直接 `v-model` しており、入力した瞬間に反映される。キャンセルしても内容は戻らない。

```html
<!-- ContextDialog.vue — settings に直接バインド -->
<textarea v-model="settings.systemPrompt" />
```

**影響:** 設定ダイアログ間で一貫性がなく、ユーザーが混乱する。

---

### W-U5: 空状態のヒントが見えない
ノードが0個の時に表示されるヒントテキストの opacity が `0.18` で、実用的な視認性に欠ける。

```html
fill="rgba(255,255,255,0.18)"
```

---

### W-U6: MCP ノードが機能しない
MCP ダイアログを開いても、MCPサーバーへの実際の接続・実行は実装されていない（`useSettings` がプレースホルダーのみ）。

---

## パラダイム（Paradigm）

### W-P1: 「三体らしさ」が動作に反映されていない
三体問題の本質（重力的相互作用・不安定な軌道・カオス）はアニメーションの比喩として使われているが、ノード間の位置関係や組み合わせがAIの挙動に影響するロジックが存在しない。

**あるべき姿:**
- Chat + Context を両方置くと、Chatが自動でシステムプロンプトを参照する
- MCP + Chat の距離がMCPツールの使用優先度に影響する
- 3つ揃った時に初めて「三体共鳴」として何らかの創発的動作が発生する

---

### W-P2: 各ノードの「役割の差」がUIから読み取れない
Chat・MCP・Context の3つは概念的には全く異なるが、現状はどれも「クリックするとダイアログが開く」だけ。ユーザーが3者の違いを発見する手がかりがない。

---

### W-P3: Voice（重心）の特権性が説明されていない
Voice が常に重心（中央）に固定されている設計意図（= 全ての会話はVoiceを起点にするという哲学）がUIから読み取れない。ただ動かせない点があるだけに見える。
